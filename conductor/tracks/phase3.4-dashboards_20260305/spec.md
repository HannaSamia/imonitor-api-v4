# Specification: Phase 3.4 — Dashboards, RotatingDashboard & DataAnalysis

## Summary

Migrate the Dashboard, RotatingDashboard, and DataAnalysis modules from Express.js v3 to NestJS v4. This covers ~31 endpoints across 3 modules and 11 database entities, enabling users to create, share, and rotate dashboards composed of widget builders and charts, and build data analysis views linking reports and charts.

## Context

iMonitor is an enterprise telecom monitoring API. Phase 3.4 builds the visualization layer on top of the already-migrated reporting infrastructure (Reports, WidgetBuilder, QBE from Phases 3.3.1–3.3.2). Dashboards aggregate widget builders and their charts into customizable layouts; RotatingDashboards cycle through multiple dashboards on a timer; DataAnalysis links reports and charts for trend analysis.

## User Story

As a NOC engineer or management stakeholder, I want to create, share, and rotate dashboards composed of widget builders and charts, and build data analysis views linking reports and charts, so that I can monitor network health and analyze trends through customizable visual layouts.

## Out of Scope

- Socket.IO real-time dashboard updates (Phase 4)
- Dashboard export to PDF/image (handled by existing export system)

- E2E tests against live database (Phase 5)

## Technical Notes

- Follow same patterns as WidgetBuilder/Reports: PrivilegeGuard on controllers, DTOs with class-validator + @ApiProperty, TypeORM object syntax
- Dashboard `options` and RotatingDashboard `dashboardIds` are `longtext` (JSON strings) — store as string, parse in application
- Join tables (dashboard_widget_builder, dashboard_chart, data_analysis_chart, data_analysis_report) use composite PKs with CASCADE deletes
- CoreDashboardError uses `int(11) AUTO_INCREMENT` PK (different from UUID pattern used by other entities)
- Cross-module references to WidgetBuilder/Reports entities via TypeORM repos (same DB), not events
