## Summary

Migrate all authentication and user management functionality from v3 (Express.js + InversifyJS) to v4 (NestJS). This covers 25 endpoints across `AuthController` (login, logout, token refresh, heartbeat, access check) and `UsersController` (CRUD, password management, privileges, side menu, settings, theme).

## Context

Phase 1 scaffolded the NestJS project with TypeORM entities, legacy DB modules, Redis, clustering, and logging. Phase 2 built the core architecture: SharedModule (DTOs, utilities), auth guards (JWT, Roles, Privilege, ApiKey), response/request interceptors, exception filter, event system, and validation pipe. The JWT guard skeleton and role/privilege guards from Phase 2 need to be wired to real service logic.

Reference: [Phase 3.1 v3 Analysis](../../docs/phase3.1-auth-users-analysis.md)

## User Story

As a system administrator, I want to authenticate users, manage accounts, and control role-based access so that the v4 API provides identical auth behavior to v3.

## Acceptance Criteria

- [ ] All 6 auth endpoints (`/api/v1/auth/*`) return identical responses to v3
- [ ] All 19 user endpoints (`/api/v1/users/*`) return identical responses to v3
- [ ] JWT generation with configurable expiry from `core_sys_config`
- [ ] Refresh token rotation (single-use, invalidated on logout)
- [ ] `keepLogin` bypass for token expiry in JWT guard
- [ ] Dynamic authorization via `core_minimum_privileges` table
- [ ] Role hierarchy enforcement (superadmin > admin > superuser > user > N/A)
- [ ] User registration with default privilege seeding (N/A role for all modules)
- [ ] Password hashing with bcrypt, change/reset flows
- [ ] Hierarchical privilege tree (recursive module/role mapping)
- [ ] Side menu filtering by user privileges
- [ ] Module settings and system configuration endpoints
- [ ] Soft delete with audit trail (deletedBy, deletedOn)
- [ ] Unit tests for AuthService and UsersService
- [ ] Swagger decorators on all endpoints
