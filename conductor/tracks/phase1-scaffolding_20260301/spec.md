# Specification: Phase 1 — Project Scaffolding & Infrastructure

**Track ID:** phase1-scaffolding_20260301
**Type:** Feature
**Created:** 2026-03-01
**Status:** Draft

## Summary

Scaffold the NestJS v4 project with all foundational infrastructure: TypeORM entities for iMonitorV3_1 (from db.sql), legacy database modules for iMonitorData/EtlV3_2/Presto, Redis module, Node.js clustering with sticky sessions, Winston logger with daily rotation, and a basic auth guard skeleton.

## Context

This is the first implementation phase of the iMonitor v3→v4 migration (see MIGRATION.md). The v3 API is an Express.js + InversifyJS + TSOA application with 150+ endpoints across 25 modules, connecting to 3 MySQL databases + Redis + Presto. This track establishes the entire infrastructure foundation that all subsequent feature module tracks will build upon.

## User Story

As a developer, I want a fully scaffolded NestJS project with all database connections and infrastructure so that I can start migrating feature modules.

## Acceptance Criteria

- [ ] NestJS project scaffolded with @nestjs/config + Joi env validation matching all v3 env variables
- [ ] TypeORM entities for all 51 iMonitorV3_1 tables matching db.sql exactly (every column, type, index, FK, constraint)
- [ ] TypeORM migration file that can recreate the full iMonitorV3_1 schema on an empty database
- [ ] LegacyDataDbModule — mysql2 connection pool for iMonitorData with v3's dual-pool pattern (main + limited)
- [ ] LegacyEtlDbModule — mysql2 connection pool for EtlV3_2
- [ ] LegacyPrestoModule — Presto client wrapper for bill run queries
- [ ] Redis module (ioredis) configured for caching, rate limiting, and Socket.IO adapter readiness
- [ ] Node.js clustering with @socket.io/sticky round-robin, auto-restart dead workers, graceful shutdown
- [ ] Winston logger with daily rotation (error.log, combined.log, emergency.log), structured JSON, correlation IDs
- [ ] Basic auth guard skeleton (JwtAuthGuard placeholder, not fully implemented)

## Dependencies

- `./imonitor-v3-api/db.sql` — Schema source of truth for TypeORM entities
- `./imonitor-v3-api/src/` — Reference code for database connection logic, Redis config, cluster setup, logger config

## Out of Scope

- Feature module controllers, services, or DTOs (Phase 2+)
- Socket.IO gateway implementations (Phase 4)
- Full auth guard/roles guard implementation (Phase 2)
- Unit or E2E tests (Phase 5, though smoke tests are acceptable)

## Technical Notes

- Database: MariaDB 11.3.2 — use `type: 'mariadb'` in TypeORM config
- `synchronize: false` always — never auto-sync schema
- Dual MySQL pool pattern from v3: main pool (DB_USER) + limited pool (DB_LIMIT_USER), both 15 connections
- .env variables must match v3 exactly: DB_HOST, DB_USER, DB_PASSWORD, DB_PORT, DB_LIMIT_USER, DB_LIMIT_PASSWORD, coreDbName, dataDbName, etlDbName, PORT, CPUS, JWT_KEY, MAIL_*, REDIS_*, NB_OF_REQUESTS, RATE_LIMIT_DURATION_SEC, RATE_BLOCK_DURATION
- Cluster: use same pattern as v3 app.ts — setupMaster with round-robin, fork CPUS workers, auto-restart on exit
- Logger: match v3's level hierarchy (emerg, error, warn, info, http, debug), same file rotation config
